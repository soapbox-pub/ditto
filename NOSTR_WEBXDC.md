# Nostr Webxdc

## Abstract

This NIP defines a protocol for sharing and running [webxdc](https://webxdc.org/) mini apps over Nostr. Webxdc apps are sandboxed HTML5 applications distributed as `.xdc` (ZIP) files that communicate via shared state updates. Rather than introducing dedicated event kinds for apps or sessions, webxdc files are attached to regular Nostr events using `imeta` tags (NIP-92), and state synchronization is coordinated through a UUID.

## Motivation

Webxdc provides a compelling model for peer-to-peer mini apps: zero internet access from within the app, no tracking, no platforms. However, webxdc currently relies on chat messenger implementations (Delta Chat, etc.) for distribution and state synchronization. By bridging webxdc to Nostr, we enable:

- **Permissionless app distribution**: Anyone can attach a webxdc app to any Nostr event.
- **Open collaboration**: Any Nostr user can interact with a webxdc by referencing its UUID.
- **Decentralized state sync**: App updates flow through Nostr relays, enabling multi-user collaboration without centralized infrastructure.
- **No new event kinds for distribution**: Webxdc apps are simply file attachments, not a separate event type.

## Webxdc Attachment

A webxdc app is attached to any event (typically kind `1`) by including the `.xdc` file URL in the event content and adding an `imeta` tag with the MIME type `application/vnd.webxdc+zip`.

### The `webxdc` Property

The `imeta` tag supports an additional property, `webxdc`, whose value is a UUID (v4) generated by the event author. This UUID serves as the coordination identifier for state updates.

- If the `webxdc` property is **present**, the app is **stateful**. Other users can send state updates by referencing this UUID in an `i` tag.
- If the `webxdc` property is **absent**, the app is **stateless**. No state synchronization occurs; the app runs in isolation for each user.

### Stateful Example

A kind `1` note with an attached stateful webxdc app:

```json
{
  "kind": 1,
  "content": "Let's play chess! https://blossom.example.com/abc123.xdc",
  "tags": [
    ["imeta",
      "url https://blossom.example.com/abc123.xdc",
      "m application/vnd.webxdc+zip",
      "x a1b2c3d4e5f6...",
      "webxdc 9b1deb4d-3b7d-4bad-9bdd-2b0d7b3dcb6d"
    ]
  ]
}
```

### Stateless Example

A kind `1` note with an attached stateless webxdc app (no `webxdc` property):

```json
{
  "kind": 1,
  "content": "Try this calculator app https://blossom.example.com/calc.xdc",
  "tags": [
    ["imeta",
      "url https://blossom.example.com/calc.xdc",
      "m application/vnd.webxdc+zip",
      "x f7e8d9c0b1a2..."
    ]
  ]
}
```

### NIP-94 File Metadata (Alternative)

A webxdc app MAY also be published as a kind `1063` (NIP-94) file metadata event. This is useful for file-sharing and indexing contexts where the webxdc is the primary content rather than an attachment.

The same `webxdc` property applies: include it in the tags to make the app stateful, or omit it for a stateless app.

```json
{
  "kind": 1063,
  "content": "A collaborative chess game. Play with friends over Nostr!",
  "tags": [
    ["url", "https://blossom.example.com/abc123.xdc"],
    ["m", "application/vnd.webxdc+zip"],
    ["x", "a1b2c3d4e5f6..."],
    ["alt", "Webxdc app: Chess"],
    ["webxdc", "9b1deb4d-3b7d-4bad-9bdd-2b0d7b3dcb6d"]
  ]
}
```

Any method that involves `imeta` tags or NIP-94 file metadata tags is suitable for attaching webxdc apps.

## State Updates

### Kind `4079`: Webxdc State Update (Regular)

A regular event that carries a state update from one participant to all others interacting with the same webxdc instance. This maps directly to the webxdc `sendUpdate()` API. Updates are ordered by their `created_at` timestamp and assigned serial numbers by the client.

**Required Tags:**

| Tag   | Description                                                  |
|-------|--------------------------------------------------------------|
| `i`   | The UUID from the `webxdc` property of the originating event |
| `alt` | NIP-31 human-readable description                            |

**Optional Tags:**

| Tag        | Description                                      |
|------------|--------------------------------------------------|
| `info`     | Short info message (shown in feed, max ~50 chars) |
| `document` | Document name being edited                        |
| `summary`  | Short summary text (e.g. "8 votes")               |

**Content:** JSON-serialized payload from `sendUpdate()`.

```json
{
  "kind": 4079,
  "content": "{\"move\":\"e2e4\",\"player\":\"white\"}",
  "tags": [
    ["i", "9b1deb4d-3b7d-4bad-9bdd-2b0d7b3dcb6d"],
    ["alt", "Webxdc update"],
    ["info", "White played e2-e4"]
  ]
}
```

## Realtime Channel

### Kind `21567`: Webxdc Realtime Data (Ephemeral)

An ephemeral event that carries realtime data between peers interacting with the same webxdc instance. This maps directly to the webxdc [`joinRealtimeChannel`](https://webxdc.org/docs/spec/joinRealtimeChannel.html) API. Ephemeral events (kind 20000–29999) are not stored by relays — they are only forwarded to clients with active subscriptions.

**Required Tags:**

| Tag | Description |
|-----|-------------|
| `i` | The UUID from the `webxdc` property of the originating event |

**Content:** Base64-encoded binary payload from `realtimeChannel.send()`. The raw `Uint8Array` (max 128,000 bytes) is encoded to a base64 string for transport in the JSON event.

```json
{
  "kind": 21567,
  "content": "SGVsbG8gZnJvbSBucHViMWFiYy4uLg==",
  "tags": [
    ["i", "9b1deb4d-3b7d-4bad-9bdd-2b0d7b3dcb6d"]
  ]
}
```

## Protocol Flow

1. **Attach App**: A user uploads their `.xdc` file to a Blossom server and publishes an event (e.g. kind `1`) with the file URL in content and an `imeta` tag with MIME type `application/vnd.webxdc+zip`. If the app is stateful, the `imeta` includes a `webxdc` property with a new UUID.
2. **Run App**: A client seeing the event detects the webxdc `imeta` tag, downloads the `.xdc` file, extracts it, and runs `index.html` in a sandboxed iframe.
3. **Send Updates**: When the app calls `sendUpdate()`, the client publishes a kind `4079` event with an `i` tag containing the UUID.
4. **Receive Updates**: The client subscribes to kind `4079` events with `#i` matching the UUID and delivers them to the app via `setUpdateListener()`.
5. **Realtime Channel**: When the app calls `joinRealtimeChannel()`, the client subscribes to kind `21567` events with `#i` matching the UUID. Calls to `realtimeChannel.send()` publish ephemeral kind `21567` events; incoming events are decoded and delivered via `realtimeChannel.setListener()`. Calling `realtimeChannel.leave()` closes the subscription.
6. **Discovery**: To find the original `.xdc` file for a given UUID, clients query for events containing an `imeta` tag with the matching `webxdc` property.
7. **Identity**: `selfAddr` maps to the user's Nostr pubkey (npub). `selfName` maps to the user's kind 0 display name.

## Security Considerations

- Webxdc apps MUST be run in sandboxed iframes with no network access (matching the webxdc spec).
- The iframe sandbox attribute should use: `allow-scripts allow-popups allow-popups-to-escape-sandbox`.
- All communication between the iframe and parent happens via `postMessage`.
- The CSP for the iframe should block all network access (`connect-src 'none'`).
- State updates should be filtered by the `i` tag matching the UUID to prevent cross-instance contamination.
- Clients SHOULD verify the `.xdc` file hash (`x` property) against the downloaded file before running it.
